# Use Case 20: Extract fork into a standalone GitHub repo

## Summary
The current `HaroldHormaechea/yt-dlp-ui` is a GitHub fork of `yt-dlp/yt-dlp` whose Python tree is never executed at runtime — `scripts/fetch-yt-dlp.sh` already pulls a SHA-pinned prebuilt binary from upstream's GitHub Releases at build/CI time. The fork structure has accumulated friction proportional to project complexity: upstream's README + LICENSE define first-impression identity, ~50 MB of Python source ships in every clone, and recent fixes (zizmor.yml ignores for `release-upstream.yml`, actionlint.yml SC2086/SC2129 ignores for autogenerated `release.yml`, path-filter narrowing on `core.yml` + `quick-test.yml`, the deferred `public.key → scripts/keys/yt-dlp.asc` rename UC 17 couldn't perform) exist solely to cope with the fork. The migration: **rename the existing fork to `HaroldHormaechea/yt-dlp-ui-fork-archive` and archive it; create a fresh `HaroldHormaechea/yt-dlp-ui` repo with only project-owned files; fresh-init git history (no continuity); drop `release-upstream.yml` and let cargo-dist regenerate `release.yml` cleanly; drop all upstream ancillary files (`.gitignore`, `.editorconfig`, `.gitattributes`, `.github/banner.svg`, `.github/FUNDING.yml`, `.github/ISSUE_TEMPLATE/`, `.github/PULL_REQUEST_TEMPLATE.md`) and recreate any the project actually needs**. The bundled-binary integration is unchanged because it was already binary-only via the existing fetch scripts. PROJECT_BRIEF.md's "Trust boundaries / read-only zone" section becomes obsolete.

## Potential Pitfalls & Open Questions
- **Edge case** — cargo-dist regeneration risk: running `dist generate --mode ci` on the new repo without `release-upstream.yml` may shift the autogenerated `release.yml` content (cargo-dist's templates change between versions; the existing `release.yml` was generated at v0.31.0 with the upstream-rename trick). Verify in the worktree-first step that the regenerated `release.yml` references the package-* trio correctly and the `dist-workspace.toml` `include` paths still resolve.
- **Edge case** — Existing GitHub Actions cache entries (the per-arch ffmpeg cache from UC 17's `package-dmg.yml`) live on the old fork; the new repo starts fresh. First release on the new repo pays full ffmpeg-build time per arch (~30 min × 2 macOS arches, one-time).
- **Edge case** — Toolchain expectations on the new repo's first CI run: `i-slint-backend-testing` dev-dep, `cargo-dist`, `cargo-deny`, `cargo-llvm-cov`, `bats`. None depend on the upstream tree, but the new `Rust CI` workflow needs to install them. The workflow definition is part of this UC.
- **Risk** — Loss of fork-relationship metadata on github.com may affect Dependabot's view, GitHub Insights metrics, and any third-party tools that rely on the fork link. None are load-bearing.
- **Risk** — `LICENSE` at repo root changes from upstream's Unlicense to PolyForm Noncommercial 1.0.0 — a *user-visible licensing change* on the project's GitHub landing page. The bundled `yt-dlp` binary's Unlicense terms continue to ship at install time per UC 17's pattern. Worth a one-line note in `THREATS.md` documenting the licensing transition for posterity.
- **Risk** — A new project-specific `.gitignore` is a day-one need (cargo build artifacts, IDE state, .env files). The dev team writes it as part of the migration and it is one of the few "recreated" ancillary files.
- **Assumption** — cargo-dist still autogenerates `release.yml` on the new repo. The `dist-workspace.toml` and `release.yml` integration are preserved.

## Original Description
Extract this fork into a standalone GitHub repo. The current repo is a fork of yt-dlp/yt-dlp which we never use at runtime (we bundle the prebuilt binary via fetch-yt-dlp.sh). The fork carries upstream's README.md / LICENSE at repo root (giving wrong first-impression identity), ships ~50 MB of Python code we never execute, and forces a "trust boundaries / read-only zone" workaround pattern that has cost the project visible friction (zizmor.yml ignores for release-upstream.yml, actionlint.yml SC2086/SC2129 ignores for autogenerated release.yml, path-filter narrowing on core.yml + quick-test.yml, the deferred public.key → scripts/keys/yt-dlp.asc rename UC 17 couldn't do, etc.). Migration plan: prune upstream tree, set up the new repo with our README/LICENSE/CI at root, verify in a worktree first, then push to a fresh GitHub repo + archive the old fork. The architecture is already correct (yt-dlp is a binary dependency, not a code dependency); the fork was just the wrong vehicle for that intent.

## Clarifications
- Q: Repo name and GitHub ownership for the new standalone repo?
  A: Reuse `HaroldHormaechea/yt-dlp-ui`. Rename the current fork to `HaroldHormaechea/yt-dlp-ui-fork-archive` first, then create the new repo at the freed name. Same URL shape; minimum disruption to anyone bookmarking the project.
- Q: Git history strategy for the new repo?
  A: Fresh init — no history continuity. Squash everything into one initial commit on the new repo. Cleanest start; loses commit-by-commit attribution.
- Q: `release-upstream.yml` on the new repo?
  A: Drop it. Re-run `dist generate --mode ci` on the new repo without the upstream-rename trick — `release.yml` regenerates at the natural path. Verify in the worktree-first step that the regenerated workflow references the package-* trio correctly.
- Q: Upstream ancillary files (`.gitignore`, `.editorconfig`, `.github/ISSUE_TEMPLATE/`, `.github/FUNDING.yml`, `.github/PULL_REQUEST_TEMPLATE.md`, `.github/banner.svg`) — keep, drop, or recreate?
  A: Drop all; recreate as needed. Don't migrate upstream's templates / configs. Add project-appropriate equivalents only when the project actually needs them (e.g., a day-one project-specific `.gitignore`).
